# 메멘토애니 검증 워크플로 (Verification Workflow)

> 작성: 2026-04-12
> 영감: leceipts (github.com/0oooooooo0)
> 목적: AI 코딩 시 "lying about done" 방지. "고쳤습니다"는 금지, 검증 결과만 진실.

---

## 핵심 원칙

### 1. 검증 안 한 건 미검증으로 명시
- "고쳤습니다" / "정상 작동합니다" / "통과했습니다" — **이런 표현 단독 사용 금지**
- 반드시 어느 검증 레벨까지 도달했는지 명시
- 검증 안 한 항목은 ⚠️ 미검증 표시 (✅ 가짜 표시 금지)

### 2. 검증은 가능한 최대한 자동화
- typecheck / build / lint / DB 컬럼 / 마이그레이션 / import 연결 — 모두 스크립트
- 자동 가능한 건 절대 수동으로 "확인했다"고 말하지 않음

### 3. 잔여 위험은 자발적으로 노출
- 마이그레이션 미실행
- 시각적 미테스트
- API curl 미실행
- 본인 계정으로 직접 확인 안 됨
→ 모두 명시. 숨기면 안 됨.

---

## 검증 레벨 (L0 ~ L5)

| 레벨 | 의미 | 자동/수동 | 신뢰도 | 통과 조건 |
|---|---|---|---|---|
| **L0_미검증** | 코드만 작성 | - | 0% | 없음 |
| **L1_빌드** | next build 통과 | 자동 | 30% | exit 0 |
| **L2_타입+빌드** | typecheck + build | 자동 | 50% | 둘 다 exit 0 |
| **L3_정적전수** | L2 + lint + import 연결 + DB 스키마 | 자동 | 70% | scripts/verify-all 통과 |
| **L4_API응답** | L3 + curl/fetch 응답 확인 | 수동/자동 | 85% | API 200 + 예상 응답 |
| **L5_E2E** | L4 + 실제 UI에서 동작 | 수동 | 100% | 본인/사용자 시각 확인 |

**원칙**: 작업 종류에 따라 도달 가능한 최대 레벨이 다름.
- 순수 코드 리팩토링 → L3 가능
- API 추가 → L4 가능 (서버 실행 시)
- UI 변경 → L5는 사용자만 가능 → L3까지 자동 + "L5 미검증" 명시

## 강제 출력 형식 (작업 끝마다)

```markdown
## 🔍 Verification Report

### 1. 작업 요약
[무엇을 왜 했는지 1-2문장]

### 2. Change Map
- [path/to/file.ts:10-25] 변경 내용 한 줄
- [path/to/new-file.ts] 신규: 목적
- [migration.sql] 신규: DB 변경 내용

### 3. Automated Checks
| Check | Result | Detail |
|---|---|---|
| typecheck (`tsc --noEmit`) | ✅ / ❌ / ⏭️ | (실패 시 에러 인용) |
| build (`next build`) | ✅ / ❌ | |
| lint (`eslint`) | ✅ / ❌ | (warning은 별도 표기) |
| Import 연결 | ✅ / ⚠️ | (신규 파일이 import 안 되면 ⚠️) |
| DB 스키마 일치 | ✅ / ⚠️ / ❌ | (마이그레이션 미실행 시 ⚠️) |
| 미실행 마이그레이션 | None / N개 | |

### 4. Manual Verification
| 항목 | 방식 | 결과 |
|---|---|---|
| API 응답 (예: POST /api/...) | curl 미실행 | ⚠️ 미검증 |
| UI 시각 동작 | 미테스트 | ⚠️ 미검증 |
| DB 데이터 일관성 | (필요 시) | ⚠️ 미검증 |

### 5. Verification Level
**달성: Lx_xxx**
- L1, L2, L3 — 자동 검증 통과
- L4, L5 — 미검증 (사용자 확인 필요 / 마이그레이션 실행 필요)

### 6. Remaining Risk
- ❗ 위험도 높은 미검증 항목
- ⚠️ 사용자가 직접 해야 할 후속 작업
- 📝 권장 검증 절차 (단계별)

### 7. Confidence Statement
**"이 작업은 [Lx]까지만 검증되었습니다. 그 이상은 보장 못 합니다."**
```

## 거짓말 방지 규칙 (Lying Patterns Blocklist)

### ❌ 금지 표현 (단독 사용 시)
- "고쳤습니다"
- "수정 완료되었습니다"
- "정상 작동합니다"
- "통과했습니다" (출력 인용 없이)
- "테스트 통과" (어떤 테스트인지 안 쓰면)
- "문제 없습니다"
- "모두 확인했습니다"
- "에러 없습니다"

### ✅ 권장 표현
- "코드 변경 완료. typecheck/build 통과(L2). DB 마이그레이션 미실행(L3 미달)"
- "API 라우트 작성 완료. 빌드 통과. curl 미실행이라 응답 검증은 사용자 필요(L3 → L4 미달)"
- "UI 컴포넌트 작성. 빌드 통과. 시각 검증은 사용자만 가능(L3 → L5 미달)"

### 자체 검열 체크리스트 (작업 끝낼 때마다 자문)
1. "고쳤다"고 했는데 build 안 돌렸나?
2. "통과했다"고 했는데 어떤 명령어를 어떤 결과로?
3. "정상 작동"이라 했는데 실제 호출해봤나?
4. DB 변경 있는데 마이그레이션 실행했나?
5. UI 변경 있는데 사용자 시각 확인이 필요한 거 아닌가?
6. import 추가했는데 정말 어디서 쓰이나?

하나라도 NO면 그 항목은 ⚠️ 미검증.

## 자동 검증 도구

### 1. `npm run verify` — 전체 검증 (verify-all)
- typecheck
- build
- lint
- import 연결 확인
- DB 스키마 확인 (Supabase URL 있을 때만)
- 미실행 마이그레이션 감지

### 2. `npm run verify:db` — DB 스키마만
- Supabase 환경변수 있으면 information_schema 쿼리
- 컬럼/제약/인덱스 존재 확인
- 코드 vs DB 불일치 감지

### 3. `npm run verify:migrations` — 미실행 마이그레이션 감지
- supabase/migrations/ 디렉토리 vs 실제 DB 적용 상태
- 미실행 SQL 파일 목록 출력

### 4. `npm run verify:imports` — Import 그래프 분석
- 신규 파일이 어디서 import되는지
- 죽은 코드(import 0개) 감지
- 순환 import 감지

### 5. `/verify` Claude 슬래시 커맨드
- 위 4개 자동 실행
- 결과를 5섹션 보고서로 정리
- 검증 레벨 자동 산정

## 작업 종류별 권장 검증 레벨

| 작업 종류 | 최소 자동 | 사용자 확인 필요 | 권장 표시 |
|---|---|---|---|
| 타입 정의 추가 | L2 | X | "L2 통과" |
| 순수 함수 추가 | L3 | X | "L3 통과 (단위 테스트 미작성)" |
| API 라우트 추가 | L3 | curl 또는 UI | "L3 통과, L4 사용자 확인 필요" |
| UI 컴포넌트 추가 | L3 | 시각 | "L3 통과, L5 사용자 확인 필요" |
| DB 마이그레이션 작성 | L1 (SQL 파싱만) | 실행 | "L1 통과, 마이그레이션 미실행" |
| 라이브러리/유틸 리팩토링 | L3 | X | "L3 통과" |
| 환경변수/설정 변경 | L1 | 재배포 | "L1 통과, 재배포 필요" |

## 메모리 시스템 통합

검증 위반 패턴은 `memory/feedback_lying_patterns.md`에 누적.
다음 세션이 자동으로 회피.

### 누적 항목
- "고쳤습니다"라고만 말하고 build 안 돌렸을 때 → 패턴 기록
- 마이그레이션 작성 후 실행 안 했는데 "완료"라고 말한 경우 → 패턴 기록
- "정상 작동"이라 한 후 사용자가 버그 발견 → 패턴 기록

### 다음 세션이 학습할 내용
- 작업 시작 시 `memory/verification_workflow.md` 자동 로드
- "고쳤다" 같은 금지 표현 사용 직전 자체 검열
- 검증 레벨 명시 없이 완료 보고 금지

## 잔여 위험 표시 표준

미실행 마이그레이션, 미검증 시각, 미실행 API 호출 등은 표준 표기로:

```markdown
### ⚠️ Remaining Risk

| 위험 | 영향 | 해결 방법 | 담당 |
|---|---|---|---|
| supabase/migrations/XXX.sql 미실행 | 발송 시 CHECK 위반 | Supabase Dashboard → SQL Editor | 사용자 |
| UI 시각 회귀 미테스트 | 톤/배지 잘못 표시 가능 | 본인 계정으로 5분 클릭 테스트 | 사용자 |
| Realtime 채널 활성화 미확인 | Realtime 토스트 미작동 | Supabase → Database → Replication | 사용자 |
```

## 위반 시 패널티

작업이 끝났는데 Verification Report 없이 "완료"라고 말하면:
- 사용자가 불만 표시 가능 ("검증했어?")
- 같은 패턴 반복 → memory/feedback_lying_patterns.md에 누적
- 다음 세션이 자동으로 더 까다롭게 검증
