## PART 1: 에이전트 간 토론 -- 의견 충돌과 판정

### 토론 1: agent.ts Service Role 취약점 심각도

**2번 (백엔드)**: "agent.ts가 Service Role Key로 RLS를 완전 우회하면서 user_id 필터가 누락된 쿼리가 있다. 전체 사용자 데이터에 접근 가능한 Critical 취약점이다."

**7번 (비판적 사고) 검증 결과**: agent.ts 455~467행을 확인. 실제로는 `agentSupabase.from("pets").select(...).eq("user_id", userId)` 형태로 user_id 필터가 **존재한다**. 다만 이 userId가 클라이언트에서 전달받은 값이 아니라 `getAuthUser()`로 서버에서 추출한 값이므로 조작 불가. **2번의 "전체 데이터 접근 가능"은 과장**. 단, Service Role 사용 자체는 RLS 우회이므로 원칙적으로는 위험 요소가 맞다.

**판정**: High (Critical → 하향). 현재 구현에서 실질적 익스플로잇 경로는 제한적이나, 향후 코드 수정 시 user_id 필터를 빠뜨리면 즉시 Critical이 되므로 구조적 리스크로 인지 필요.

### 토론 2: display:none 탭 숨김 패턴의 메모리 영향

**1번 (프론트엔드)**: "mountedTabs로 방문한 탭을 display:none으로 숨기면 8개 탭 모두 DOM에 유지되어 메모리 누수다. P1 수준."

**3번 (UX)**: "맞지만, 탭 전환 시 이전 스크롤 위치를 기억하고 데이터 재로딩이 없어서 UX는 오히려 좋다. Low 수준."

**8번 (PM)**: "SPA에서 display:none은 일반적인 트레이드오프다. 메모리 2배 vs 사용자 경험 개선. 모바일에서 문제가 될 수 있지만 당장은 아니다."

**7번 (비판적 사고) 판정**: page.tsx 405~441행 확인. `mountedTabs` 패턴은 의도된 설계이며, React의 `<Suspense>` + `keep-alive` 패턴의 수동 구현이다. 8개 탭이 모두 마운트되어도 이미지 lazy loading이 적용되어 있다면 메모리 이슈는 제한적. **P2로 조정** (P1 → 하향). 다만 AdminPage가 포함되면 관리자 데이터가 일반 사용자 DOM에 남을 수 있으므로 관리자 탭만은 조건부 언마운트 고려.

### 토론 3: Rate Limiter 인메모리 문제

**2번 (백엔드)**: "인메모리 Map 기반 Rate Limiter는 Vercel Serverless에서 완전히 무효하다. Critical."

**5번 (QA)**: "같은 의견. 서버리스에서 Map은 인스턴스마다 독립이라 공유 안 됨."

**8번 (PM)**: "High. DAU가 낮은 초기에는 Vercel이 인스턴스를 재사용하는 경우가 많아 완전 무효는 아니다. 하지만 스케일링되면 무의미해진다."

**7번 판정**: rate-limit.ts 14~17행과 224행 확인. `ipRequestCounts`, `ipBlockList`는 메모리 기반이지만, `checkDailyUsageDB` 함수명으로 DB 기반 일일 체크가 별도로 존재한다. **이중 구조**: IP 레벨 rate limit은 메모리(무효), 사용자 일일 제한은 DB(유효). **High로 유지하되 "완전 무효"는 과장**. IP 기반 DDoS 방어만 실질적으로 안 되는 것이며, 과금 연동된 일일 사용량 제한은 DB 기반이므로 작동한다. 단, OpenAI API 비용 폭주 방어를 위해 Upstash Redis 도입은 필요.

### 토론 4: 프롬프트 인젝션 방어 수준

**6번 (AI/프롬프트)**: "유니코드 정규화, 제로폭 문자, 다국어 혼합 등 우회 벡터가 있다. P0."

**2번 (백엔드)**: "sanitizeInput이 XSS 방어용이지 AI 보안과는 무관하다."

**7번 판정**: chat/route.ts의 sanitizeInput 함수와 `<user_input>` 태그 기반 격리를 확인. GPT-4o-mini는 `<user_input>` 태그를 시스템 프롬프트 경계로 인식하지만, 모델 수준의 보장은 없다. **6번이 맞다. P0은 합당**. 다만 "즉시 서비스 중단 수준"은 아니다. AI 인젝션으로 할 수 있는 최악은 부적절한 응답 생성이며, DB 접근이나 시스템 명령 실행은 불가. 실질적 임팩트를 고려하면 **High**가 적절 (P0 → 하향). sanitizeInput에 유니코드 정규화(`NFC`) + 제로폭 문자 제거 추가는 빠른 시일 내 필요.

### 토론 5: AuthContext 과도한 책임

**1번 (프론트엔드)**: "20개+ 상태가 하나의 Context에 있다. P1."

**7번 검증**: AuthContext.tsx 88~110행 확인. 실제 `useState`는 11개. 20개는 과장. 그러나 인증, 프로필, 프리미엄, 관리자, 간편모드, 온보딩까지 한 Context에 있는 건 사실. **P1 유지하되 "20개+ 상태"는 "11개 상태 + 다수의 파생 값"으로 정정**. Context 분리 자체는 PetContext가 이미 MemorialModeContext, TimelineContext로 분리된 선례가 있으므로 AuthContext도 같은 패턴 적용 가능.

### 토론 6: 온보딩 7단계 문제

**3번 (UX)**: "7단계 온보딩은 인지 부하가 크다. Critical."

**8번 (PM)**: "반려동물 서비스 특성상 펫 정보 수집이 필수다. 단계를 줄이면 핵심 데이터가 빠진다."

**7번 판정**: 온보딩은 이름, 사진, 생년월일, 종/품종, 성별, 성격, 특이사항 등을 수집한다. 이 중 필수는 이름과 종 정도. **3번이 맞다**. 필수 2~3단계 + 선택적 "나중에 입력" 전략이 적절. 7단계를 모두 강제하면 이탈률이 높다. 단, "Critical"은 과장. **High**가 적절 (서비스가 동작하지 않는 건 아니므로).

### 토론 7: chat/route.ts pet.id 소유권 검증

**2번 (백엔드)**: "chat API에서 body로 받은 pet.id가 현재 로그인 유저 소유인지 검증하지 않는다. High."

**5번 (QA)**: "pet.type도 무검증이라 프롬프트 주입 벡터가 된다. High."

**7번 검증**: chat/route.ts 769~785행 확인. `body.pet`이 클라이언트에서 통째로 전달되며, `pet.id`를 DB 조회 키로 사용하면서 소유권 검증이 없다. 또한 `pet.name`, `pet.type`, `pet.personality` 등이 AI 프롬프트에 직접 삽입된다. **2번과 5번 모두 맞다. 통합하여 Critical로 상향**. 다른 유저의 펫 데이터를 사용한 AI 대화가 가능하고, `pet.type`에 악성 프롬프트를 삽입하면 시스템 프롬프트를 오염시킬 수 있다.

## PART 3: 긍정적 평가 (7번 비판적 사고 인정)

모든 게 문제인 건 아니다. 이 프로젝트에서 **잘 된 부분**을 객관적으로 인정한다:

### 아키텍처
- **TypeScript strict 모드** 활성화, `as any` 1건, `@ts-ignore` 0건 → 타입 안전성 우수
- **중앙 타입 관리** (`types/index.ts` 899행) → 대부분의 타입이 한 곳에서 관리됨
- **Context 분리 선례** → PetContext에서 MemorialModeContext, TimelineContext 분리 완료
- **Supabase lock 데드락 방지** → `promiseLock` 패턴으로 navigator.locks 문제 우회 (기술적 깊이)

### 보안
- **다층 보안**: Rate Limiting + VPN 감지 + 프롬프트 인젝션 감지 + 위기 감지 + XSS 방지
- **보안 헤더**: CSP + X-Frame-Options + X-Content-Type-Options 설정
- **IP 기반 다중 계정 제한** + 관리자 예외 처리
- **민감 컬럼 보호 SQL** (코드는 작성됨, 실행만 하면 됨)

### AI 시스템
- **14단계 처리 파이프라인** → 위기 감지 → 감정 분석 → 장기 메모리 → 성격 매핑 → 응답 검증
- **듀얼 모드** (일상/추모) → 각 모드별 프롬프트, 감정 분석, 후속 질문 분기
- **7종 성격 x 2모드 = 14종 말투 매핑** → 차별화된 AI 경험
- **자동 타임라인** (10턴마다 대화 요약 저장) → 장기 사용자 경험

### 법적/운영
- 개인정보처리방침, 이용약관, 위치기반 약관, 커뮤니티 가이드라인 모두 존재
- 쿠키 동의 배너 구현
- 관리자 대시보드 (유저 관리, 게시물 관리, 통계)

### 코드 품질
- 모든 API 라우트에 try-catch + HTTP 상태 코드 + 사용자 친화적 에러 메시지
- useEffect cleanup 함수에서 이벤트 리스너, 타이머 정리 양호
- reactStrictMode: true 활성화
- ErrorBoundary 이중 구조 (Next.js + React 클래스 컴포넌트)

## PART 4: 최종 총평

### 한 줄 요약
**기술적 깊이는 인상적이지만, "완성"과 "작동"의 간극을 메워야 한다.**

### 상세 평가

**강점**: 1인 개발(+AI 보조)로 8개 탭, 53개 API, 듀얼 모드 AI, 14단계 처리 파이프라인까지 구현한 것은 양적으로 상당하다. TypeScript strict 모드에서 as any 1건만 사용한 것은 타입 규율이 잘 잡혀 있다는 증거다. 반려동물 추모라는 감성적 도메인에서 위기 감지, 감정 분석, 완곡 표현 처리 등 세심한 배려가 코드 곳곳에 반영되어 있다.

**약점**: "만들어놓고 안 켜놓은" 기능이 너무 많다. DB 마이그레이션 12건 미실행은 코드의 40%가 사실상 죽은 코드라는 의미다. 테스트 0건, 모니터링 0건인 상태에서 1,260줄짜리 AI 라우트를 운영하는 것은 눈 감고 운전하는 것과 같다. 결제 미연동은 수익 모델이 파워포인트에만 존재한다는 뜻이다.

**현실적 조언**:
1. 지금 가장 중요한 건 **SQL 12건 실행 + vercel.json 크론 설정**이다. 코드를 한 줄도 안 짜고 30분이면 끝나는데, 이것만으로 동작 완성도가 60% → 80%로 점프한다.
2. 결제(포트원)는 외부 계정 필요하므로 병렬로 진행하되, 최소 1주일 잡아야 한다.
3. 테스트와 모니터링은 런칭 후 1개월 내에 도입해도 된다. 완벽한 테스트 커버리지보다 **사용자 피드백이 더 가치 있는 시기**다.
4. chat/route.ts의 pet.id 소유권 검증과 chatHistory role 필터는 **코드 5줄이면 해결되는데 보안 임팩트가 크다**. 이건 오늘 안에 해야 한다.
